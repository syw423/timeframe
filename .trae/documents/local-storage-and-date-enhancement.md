# 方案：本地存储 + 日期筛选改进

## 1. 本地存储（IndexedDB）

### 目标
用户选择一次照片文件夹后，下次打开页面不再需要重新选择，自动加载已缓存的照片。

### 方案选型
**IndexedDB** — 浏览器自带的本地数据库，可存储图片等二进制大文件。主流浏览器均支持，存储空间按磁盘剩余空间动态分配（通常可达数百 MB ~ GB 级）。

### 数据存储结构

```
数据库名: TimeFrame
对象仓库: photos
主键: 自动递增 (id)
每条记录:
  { filename: string, date: string, note: string, blob: Blob, savedAt: number }
```

### 用户流程

**首次访问：**
1. 显示欢迎页 → 选择文件夹
2. 读取文件、提取 EXIF、处理备注（保持现有逻辑不变）
3. 处理完成后，将所有照片的 blob + 元数据写入 IndexedDB
4. 进入画廊

**再次访问：**
1. 页面加载 → 检查 IndexedDB 是否有数据
2. 有数据 → 隐藏欢迎页，直接从 IndexedDB 读取 → 生成 Blob URL → 进入画廊
3. 无数据（或用户清除了浏览器数据）→ 显示欢迎页 → 重新选择

**不设「更换文件夹」按钮** — 照片仓库是固定的，除非用户手动清除浏览器数据，否则不会丢失。

### 涉及改动

| 文件 | 改动 |
|---|---|
| `script.js` | 新增 IndexedDB 工具函数（`saveToIndexedDB`、`loadFromIndexedDB`） |
| | 修改页面启动逻辑：先检查 IndexedDB，有数据则跳过欢迎页 |
| | 修改文件夹选择回调：处理完成后调用 `saveToIndexedDB()` |

### 边界情况
- 用户清除浏览器数据 → IndexedDB 被清空 → 回到首次访问流程
- 文件夹中新增照片 → 不支持自动增量（无法访问文件系统），需清除浏览器数据后重新选择
- IndexedDB 配额不足 → try-catch 捕获，提示用户存储空间不足
- 大量照片 → 分批次写入，避免阻塞 UI

## 2. 日期筛选改进

### 目标
在日期选择弹窗中，只展示有照片的日期，无照片的日期不可选。

### 方案
在现有日期弹窗中新增「可用日期」区域，展示所有有照片的日期作为可点击卡片。

### 实现方式

1. **收集可用日期**
   - `const availableDates = [...new Set(allPhotoData.map(p => p.date))].sort()`
   - 存储在全局变量中

2. **弹窗 UI 改动**
   - 在「年/月/日」输入框下方新增 `#available-dates` 容器
   - 动态生成日期卡片（`div.date-chip`）
   - 每个卡片显示日期和照片数量（如 `2026-06-20 (5张)`）
   - 点击卡片 → 自动填入年/月/日输入框 → 自动执行跳转

3. **联动**
   - 切换日期（跳转/全部）时不需要刷新可用日期列表
   - 只有重新选择文件夹后才更新可用日期列表
   - 「全部」按钮保持原样

### 涉及改动

| 文件 | 改动 |
|---|---|
| `index.html` | 在日期弹窗中添加 `#available-dates` 容器 |
| `script.js` | 新增全局 `availableDates` + `buildDateChips()` 函数 |
| | 文件处理完成后调用 `buildDateChips(allPhotoData)` |
| | 点击日期芯片时自动填入并跳转 |
| `style.css` | 新增 `.date-chip` 样式（圆角标签风格） |

### 边界情况
- 只有 1 个日期 → 正常展示
- 日期很多 → 容器可滚动（`overflow-y: auto`），限制高度
- 切换「全部」后不刷新芯片列表（芯片列表只随文件夹加载变化）

---

## 实现顺序

1. 日期筛选改进（改动小，先做）
2. IndexedDB 本地存储（改动大，后做）
3. 测试验证

## 验证方式

1. 选择照片文件夹 → 刷新页面 → 应自动加载，无需重新选择
2. 日期弹窗中只显示有照片的日期
3. 点击日期芯片 → 自动跳转到该日期
4. 清除浏览器数据后刷新 → 回到欢迎页
